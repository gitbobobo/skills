# 改善代码库架构

基于 John Ousterhout《软件设计哲学》的核心思想，通过加深模块提升代码库的架构质量。

## 核心理念

**深度模块**拥有小巧的接口和强大的实现——调用者用很少的代码完成很多事。**浅层模块**的接口复杂度接近甚至超过其实现复杂度，它不提供杠杆率，只是传递复杂性。

本技能的目标是：找出浅层模块，将它们加深为深度模块。

## 词汇表

完整术语定义见 [LANGUAGE.md](LANGUAGE.md)。核心术语速查：

| 术语 | 含义 |
|------|------|
| **深度（Depth）** | 接口简单程度与实现复杂程度的比值 |
| **浅层模块** | 接口复杂度 ≈ 实现复杂度的模块 |
| **缝隙（Seam）** | 无需修改代码即可改变行为的位置 |
| **适配器（Adapter）** | 在缝隙处满足接口的具体实现 |
| **杠杆率（Leverage）** | 调用者从深度中获得的收益——每单位接口能获得更多能力 |
| **局部性（Locality）** | 维护者从深度中获得的收益——变更集中在一处 |
| **删除测试** | 删除模块后，复杂性是消失了（传递式）还是分散到 N 个调用方（有价值的） |

## 流程

### 步骤一：探索

用户描述他们想要改善的代码库区域。你需要：

1. 阅读相关源代码，理解当前架构
2. 识别该区域中的所有模块
3. 对每个模块进行**删除测试**：如果删除这个模块，复杂性会消失还是分散？
4. 标记出未通过删除测试的浅层模块——它们是加深候选

**删除测试的具体做法**：
- 想象删除该模块
- 如果调用方可以直接使用底层实现而无额外成本 → 模块是传递式的（浅层）
- 如果调用方必须各自重新实现相同逻辑 → 模块在承担有价值的职责（深层）

### 步骤二：呈现候选方案

向用户展示你找到的浅层模块：

```
## 加深候选

### [模块名称]
- **当前接口**：[描述接口暴露了什么]
- **实现复杂度**：[描述实际做了什么]
- **删除测试结果**：[传递式 / 有价值]
- **问题**：[为什么它是浅层的]

### ...
```

然后让用户选择要加深的模块。

### 步骤三：加深循环

对用户选中的每个模块，进入以下循环：

1. **呈现加深方案**——描述你计划如何加深该模块
2. **接受追问**——用户可以质疑你的方案
3. **调整并重新呈现**——根据反馈修改方案
4. 直到用户满意，执行变更

加深的策略详见 [DEEPENING.md](DEEPENING.md)。

如果涉及新接口的设计，使用 [INTERFACE-DESIGN.md](INTERFACE-DESIGN.md) 中的"设计两次"方法。

### 步骤四：架构文档

在加深过程中，如果遇到以下情况，建议用户记录架构决策：

**何时记录 ADR**：
- 决策难以逆转
- 决策令人意外（不符合常规选择）
- 存在真正的权衡

**ADR 格式**（存放在 `docs/adr/` 目录，文件名 `0001-slug.md`）：
```markdown
# [决策标题]

[1-3 句话说明决策内容和原因。可以是一段话。]
```

ADR 不需要冗长。好的 ADR 用几句话说清楚"做了什么决定、为什么"。

**何时建议创建/更新领域术语表（CONTEXT.md）**：

如果加深过程中发现团队对同一概念使用不同术语，建议创建或更新项目根目录的 `CONTEXT.md`：

```markdown
# [项目名称] 领域语言

## 术语

### [术语]
- **定义**：[一句话定义]
- **避免使用**：[容易混淆的替代词]

## 关系

- [术语A] 包含 [术语B]
- [术语C] 是 [术语D] 的 [关系]

## 有歧义的用词标记

| 上下文 | 含义 | 建议 |
|--------|------|------|
| ... | ... | 统一使用 [术语] |
```

术语表的原则：
- **有主见**——为每个概念选一个词，不要列出同义词让人选择
- **标记冲突**——如果同一词在不同上下文中含义不同，记录下来
- **定义简洁**——每条术语一句话
- **展示关系**——术语之间的包含、组合、层级关系

单仓库 vs 多仓库策略：
- 单一代码库：一个 `CONTEXT.md` 在根目录
- 多团队代码库：根目录放 `CONTEXT-MAP.md`（指向各子项目的 CONTEXT.md），每个子项目有自己的 `CONTEXT.md`

### 步骤五：生成报告

加深完成后，生成一份可视化的 HTML 架构审查报告，呈现候选模块和加深方案。报告格式详见 [HTML-REPORT.md](HTML-REPORT.md)。

## 注意事项

- 不要一次性加深所有模块——每次聚焦一个区域
- 加深可能引入新的浅层——每次变更后重新运行删除测试
- 测试策略：在加深的模块接口处写测试，而不是在旧的浅层模块上叠加测试
- 如果用户没有指定区域，从变更最频繁的目录开始
