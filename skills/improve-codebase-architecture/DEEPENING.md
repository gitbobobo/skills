# 加深策略

## 依赖分类

所有依赖分为四类，每类的测试策略不同：

### 1. 进程内依赖（In-process）
同进程内的模块间依赖。例如：工具函数、领域逻辑、数据转换。

**测试策略**：直接调用，不需要模拟。用真实实现测试。

### 2. 本地可替代依赖（Local-substitutable）
同项目内、可被替代的依赖。例如：本项目的数据库客户端、文件系统封装。

**测试策略**：通过缝隙替换为轻量适配器（如内存实现）。

### 3. 远程拥有依赖（Remote-owned / Ports & Adapters）
外部系统提供的接口，你无法控制其行为。例如：第三方 API、云服务、消息队列。

**测试策略**：定义端口（你自己的接口），为外部系统写适配器。测试中使用模拟适配器。

### 4. 真正的外部依赖（True External / Mock）
无法控制、不可替换、且没有稳定接口的外部依赖。例如：系统时间、随机数、全局状态。

**测试策略**：必须模拟（Mock）。将模拟范围限制在最小边界内。

## 加深方法

### 方法一：隐藏实现细节

最常见的加深方式。将接口中暴露的实现细节移到内部。

**浅模块示例**：
```typescript
// 调用者需要理解缓存策略、序列化方式、错误处理
function getUser(id: string): User | null {
  const cached = cache.get(`user:${id}`);
  if (cached) return JSON.parse(cached);
  const user = db.query('SELECT * FROM users WHERE id = ?', [id]);
  if (user) cache.set(`user:${id}`, JSON.stringify(user), 3600);
  return user;
}
```

**加深后**：
```typescript
// 调用者只需知道：给一个 ID，返回一个 User
function getUser(id: string): Promise<User> {
  // 缓存、序列化、错误处理全部隐藏
}
```

### 方法二：合并浅层模块

如果多个浅层模块总是被一起使用，将它们合并为一个更深的模块。

**特征**：
- 调用方总是同时导入多个模块
- 模块之间有紧密的数据流
- 单独使用某个模块没有意义

### 方法三：消除传递式抽象

如果模块只是把调用转发给另一个模块，没有增加任何价值，直接删除它。

**判断标准**：
1. 删除该模块
2. 调用方直接使用底层模块
3. 调用方的代码量没有增加 → 该模块是传递式的，应该删除

### 方法四：提取深层模块

从一个大模块中提取出真正独立的深层模块。

**注意**：提取的目标是比原模块更深，而不是更浅。如果提取出的模块是浅层的，不要提取。

## 缝隙纪律

### 内部缝隙
- 用于同一模块内的可替换部分
- 适配器在模块内部定义和使用
- 对调用者透明

### 外部缝隙
- 用于模块与外部系统的边界
- 端口（接口）由模块定义，适配器由集成层提供
- 调用者只需要知道端口

### 何时引入缝隙
- **应该**：模块需要支持多种实现（如不同存储后端）
- **应该**：需要在不修改模块的情况下改变行为（如测试替身）
- **不应该**：当前只有一个实现，且明确不会出现第二个
- **不应该**：为了让代码"更灵活"而提前抽象

## 测试策略

### 替换，而非叠加

加深模块后，测试策略应随之改变：

1. **删除旧测试**：浅层模块的单元测试在模块被加深或删除后变为负担。它们测试的是已经不存在的结构。
2. **在接口处写新测试**：在加深后的模块接口处编写测试，验证行为而非实现。
3. **不要叠加测试层**：不要在原有测试之上再叠加一层测试。用新测试替换旧测试。

### 测试金字塔的变化

加深模块会改变测试金字塔的形状：
- 浅层模块多 → 大量细粒度单元测试，但测试的是传递逻辑
- 深层模块多 → 更少的接口级测试，但每个测试覆盖更多真实行为
