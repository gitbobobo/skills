# 接口设计方法

## 设计两次

不要接受第一个浮现的接口方案。用约束驱动的方法，强制自己探索不同的设计空间。

## 三步流程

### 第一步：框定问题

在开始设计接口之前，明确：
- 这个模块要解决什么问题
- 谁是调用者
- 调用者的典型使用场景是什么

### 第二步：生成多个方案

使用"设计两次"方法：生成 3 个以上截然不同的接口方案。

每个方案从不同的约束出发：

| 方案 | 约束 | 适合场景 |
|------|------|----------|
| **最小接口** | 暴露尽可能少的方法和类型 | 接口稳定性是首要目标 |
| **最大灵活性** | 支持尽可能多的使用场景 | 需求不确定，接口可能频繁变化 |
| **优化常见路径** | 让最常见的操作最简单 | 有明确的 80/20 使用模式 |
| **端口与适配器** | 明确区分内部逻辑和外部依赖 | 需要支持多种外部系统 |

每个方案需要包含：

```markdown
## 方案 [N]：[名称]

### 接口类型
[列出暴露的类型、函数签名]

### 使用示例
[展示调用者如何使用这个接口]

### 隐藏的实现
[说明哪些复杂性被隐藏在接口之后]

### 依赖策略
[如何处理外部依赖]

### 权衡
[这个方案的优势和劣势]
```

### 第三步：对比并选择

将所有方案放在一起对比：

1. **杠杆率对比**：哪个方案让调用者写的代码最少？
2. **局部性对比**：哪个方案让变更最集中？
3. **删除测试**：如果删除每个方案的模块，调用方需要承担多少复杂性？
4. **缝隙数量**：哪个方案引入的缝隙最少且必要？

选择综合表现最好的方案。通常不是某个极端方案，而是多个方案优点的组合。

## 实践技巧

### 从调用者代码开始

先写调用者想怎么使用，再设计接口。这确保接口是为使用者设计的，而不是为实现者设计的。

```typescript
// 先写理想的调用方式
const user = await userRepo.findById('123');
const orders = await user.orders({ status: 'active' });

// 再设计支持这种调用方式的接口
```

### 用删除测试验证

设计完接口后，运行删除测试：
- 如果删除这个模块，调用方需要写多少代码来补偿？
- 需要写的代码越多 → 模块提供的杠杆率越高 → 接口设计越好

### 避免过度参数化

```typescript
// 不好：调用者需要理解所有参数
function query(opts: {
  table: string;
  where?: Record<string, any>;
  orderBy?: string;
  limit?: number;
  offset?: number;
  joins?: JoinClause[];
  // ... 更多参数
}): Promise<any[]>

// 好：为常见场景提供简单接口，高级场景用不同入口
function findById(id: string): Promise<User>;
function findWhere(conditions: Partial<User>): Promise<User[]>;
```

### 接口应该小到令人不舒服

如果接口看起来"刚好够用"，它可能还是太大了。继续问：能再移走一个参数吗？能再合并两个方法吗？接口越小，模块越深。
