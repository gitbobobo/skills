# 词汇表

## 术语定义

### 模块（Module）
一段被接口隔离的实现代码。模块通过接口与外部交互，内部实现对外不可见。

### 接口（Interface）
调用者使用模块的方式。包括函数签名、类型定义、配置选项、错误类型——任何调用者需要理解的东西。

### 实现（Implementation）
接口背后的代码。调用者不需要直接接触的部分。

### 深度（Depth）
接口简单程度与实现复杂程度的比值。**深度是接口的属性，不是实现的属性。**

- 深模块：小接口，大实现。调用者用很少的代码完成很多事。
- 浅模块：接口复杂度接近实现复杂度，甚至更高。

### 缝隙（Seam）
代码中可以改变行为而无需修改代码的位置。

- **内部缝隙**：模块内部可替换的部分（通过依赖注入、策略模式等）
- **外部缝隙**：模块与外部系统之间的边界

单个适配器只代表缝隙的一个假设——缝隙本身是模式，适配器是实现。

### 适配器（Adapter）
在缝隙处满足接口的具体实现。例如：用内存数据库替代真实数据库进行测试。

### 杠杆率（Leverage）
调用者从深度中获得的收益。接口越小，调用者需要理解的越少，同时能完成的事情越多。

### 局部性（Locality）
维护者从深度中获得的收益。变更集中在模块内部，不需要修改调用方。

## 核心原则

### 深度是接口的属性
模块的价值不在于实现有多复杂，而在于接口有多简单。一个精巧的实现如果暴露了复杂的接口，仍然是浅模块。

### 删除测试
判断模块价值的唯一标准：删除它，看复杂性去了哪里。

- 复杂性消失了 → 模块是传递式的（浅层），没有提供价值
- 复杂性分散到 N 个调用方 → 模块在承担有价值的职责（深层）

### 接口是测试面
测试应该验证模块的接口行为，而不是实现细节。在加深的模块接口处写测试。

### 一个适配器 = 一个假设的缝隙
适配器证明缝隙存在，但缝隙的真正价值在于它允许多种实现共存。如果只有一个适配器且永远不会有第二个，这个缝隙可能是不必要的抽象。

## 术语关系

```
深度
├── 杠杆率（调用者视角：接口小，能力大）
└── 局部性（维护者视角：变更集中）

模块
├── 接口（外部可见的契约）
└── 实现（隐藏的内部逻辑）

缝隙
└── 适配器（具体的替代实现）
```

## 避免使用的说法

| 避免使用 | 使用 | 原因 |
|----------|------|------|
| 组件（Component） | 模块（Module） | "组件"含义模糊，缺乏接口/实现的区分 |
| 服务（Service） | 模块（Module） | "服务"通常暗示部署边界，与深度无关 |
| API | 接口（Interface） | API 暗示远程/HTTP，接口更通用 |
| 边界（Boundary） | 缝隙（Seam） | "边界"是静态的，"缝隙"强调可替换性 |
