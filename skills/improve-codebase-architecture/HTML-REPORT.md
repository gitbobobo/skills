# HTML 报告格式

架构审查以单个自包含的 HTML 文件呈现，输出到系统临时目录。样式使用 Tailwind CDN，图表使用 Mermaid CDN。

## 页面结构

```html
<!DOCTYPE html>
<html>
<head>
  <script src="https://cdn.tailwindcss.com"></script>
  <script type="module">import mermaid from 'https://cdn.jsdelivr.net/npm/mermaid/+esm'</script>
</head>
<body class="bg-stone-50 text-stone-800 font-sans max-w-5xl mx-auto px-6 py-10">
  <!-- 报告内容 -->
</body>
</html>
```

## 页头

仓库名称、日期、紧凑的图例：

- 实线框 = 模块
- 虚线 = 缝隙
- 红色箭头 = 泄漏
- 粗黑框 = 深度模块

没有介绍段落——直接进入候选方案。

## 候选卡片

图表承担主要表达。文字精简、直白，使用词汇表中的术语（见 [LANGUAGE.md](LANGUAGE.md)），不加修饰。

每个候选是一个 `<section>` 卡片，包含：

- **标题** — 简短，命名加深操作（如"合并订单处理流水线"）
- **徽章行** — 推荐强度（`强烈推荐` 绿色 / `值得探索` 琥珀色 / `推测性` 灰色）+ 依赖分类标签（`进程内` / `本地可替换` / `端口与适配器` / `模拟`）
- **文件** — 等宽字体列表，`font-mono text-sm`
- **Before / After 图表** — 核心。两列并排，参见下方图表模式
- **问题** — 一句话。痛点在哪
- **方案** — 一句话。改变什么
- **收益** — 要点列表，每条不超过 6 个词。如"测试只需一个接口"、"定价逻辑不再泄漏"、"删除 4 个浅层包装"
- **ADR 提示**（如适用）— 琥珀色底色的一行文字

没有解释性段落。如果图表需要段落才能理解，重新画图表。

## 图表模式

选择适合候选的模式，混合使用。不要让每个图表看起来一样——多样性本身就有意义。

### Mermaid 流程图（依赖/调用流的主力）

当重点是"X 调用 Y 调用 Z，看看这团乱麻"时，使用 Mermaid `flowchart` 或 `graph`。用 Tailwind 卡片包裹，不要让它看起来像硬塞进来的。用 `classDef` 将泄漏边标红，深度模块标深色。时序图适合"之前：6 次往返；之后：1 次"。

```html
<pre class="mermaid">
flowchart LR
  A[OrderHandler] --> B[OrderValidator]
  B --> C[OrderRepo]
  C -.泄漏.-> D[PricingClient]
  classDef leak stroke:#dc2626,stroke-width:2px
  class C,D leak
</pre>
```

### 手绘框图（当 Mermaid 布局不配合时）

模块用带边框的 `<div>` 表示，箭头用内联 SVG `<line>` 或 `<path>` 绝对定位。当你想让"之后"图看起来像一个粗边框的深度模块、内部灰化时使用——Mermaid 无法渲染出正确的权重。

### 横截面（适合展示层级式浅层）

堆叠水平条带（`h-12 border-l-4`）展示调用穿过的层。之前：6 个薄层，每个什么都没做。之后：1 个粗条带，标注合并后的职责。

### 体量对比图（适合"接口和实现一样宽"）

每个模块两个矩形——一个代表接口面积，一个代表实现。之前：接口矩形和实现矩形几乎一样高（浅层）。之后：接口矩形矮，实现矩形高（深层）。

### 调用图折叠

之前：渲染为嵌套框的函数调用树。之后：同一棵树折叠为一个框，内部调用淡色显示。

## 风格指南

- 偏编辑风格，而非企业仪表盘。大量留白。标题可选衬线体（`font-serif` 搭配 stone/slate 色系效果不错）
- 克制用色：一个主色调（翡翠绿或靛蓝）+ 红色（泄漏）+ 琥珀色（警告）
- 图表高度约 320px，让 before/after 舒适地并排展示，无需滚动
- 图表中的模块标签用 `text-xs uppercase tracking-wider`——应该读起来像示意图，不像 UI
- 唯一的脚本是 Tailwind CDN 和 Mermaid ESM 导入。报告其余部分是静态的——没有应用代码，除 Mermaid 自身渲染外没有交互

## 首选推荐区

一个较大的卡片。候选名称，一句话理由，锚点链接到其卡片。仅此而已。

## 语调

平实、简洁——但架构名词和动词严格使用 [LANGUAGE.md](LANGUAGE.md) 中的术语。简洁不是偏离术语的借口。

**必须使用**：模块、接口、实现、深度、深、浅、缝隙、适配器、杠杆率、局部性。

**不得替换为**：组件、服务、单元（指代模块时）· API、签名（指代接口时）· 边界（指代缝隙时）· 层、包装器（指代模块时）。

**符合风格的表述**：
- "订单处理模块是浅层的——接口几乎等同于实现。"
- "定价逻辑跨缝隙泄漏。"
- "加深：一个接口，一个测试点。"
- "两个适配器证明缝隙的合理性：生产用 HTTP，测试用内存。"

**收益要点**用词汇表术语命名收益：*"局部性：缺陷集中在一个模块"*, *"杠杆率：一个接口，N 个调用点"*, *"接口缩小；实现吸收包装器"*。不要写 *"更易维护"* 或 *"代码更整洁"*——这些词不在词汇表中，不配占位置。

不模棱两可，不铺垫开场白，不写"值得注意的是……"。如果一句话可以变成要点，就变成要点。如果一个要点可以删掉，就删掉。如果一个术语不在 [LANGUAGE.md](LANGUAGE.md) 中，先找一个已有的替代，再考虑发明新术语。
